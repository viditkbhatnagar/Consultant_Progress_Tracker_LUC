# Backup & Disaster Recovery

> v0.1 — drafted 2026-04-26 · Last reviewed: 2026-04-26 · Owner: Engineering lead `[FILL]` + DBA `[FILL]`

## Scope

Recovery of the Team Progress Tracker database and supporting application
state in the event of accidental deletion, corruption, regional outage,
or vendor incident.

## Recovery objectives

| Metric | Target | Notes |
|---|---|---|
| **RPO** (recovery point objective) | ≤ 24 hours | Limited by Atlas snapshot cadence on the current cluster tier `[FILL]` |
| **RTO** (recovery time objective) | ≤ 8 hours for SEV-1, ≤ 24 hours for SEV-2 | Includes Atlas restore + smoke tests + comms |

These targets are aspirational until a restore drill validates them.
Recommended: first drill within 60 days of this document's adoption.

## Backups

### Database — MongoDB Atlas (Ireland)

- **Mechanism**: Atlas-managed snapshots.
- **Frequency**: Per cluster tier `[FILL]` (typical: every 6 or 24 hours).
- **Retention**: Per cluster tier `[FILL]`.
- **Encryption**: Atlas-managed at rest.
- **Geo-redundancy**: Snapshots replicate within Atlas's Ireland region by
  default; cross-region copy can be enabled for additional cost.

### Application

- The application is stateless. Render builds from `main`; recovery means
  redeploy.
- The 16 LUC program PDFs ship inside the repo at
  `client/public/program-docs/`. Source of truth: the git repo.
- Static highlight assets are regenerated by
  `npm run highlight:docs`; not backed up separately.

### Secrets

- Production secrets live in Render's env-var store.
- A printed copy of the bootstrap admin password and Atlas user password
  is stored in `[FILL: physical safe / secrets manager]`.
- See [Encryption & Key Management](04-encryption-and-key-management.md).

## Restore procedure

### Full database restore (worst case)

1. Identify the target snapshot in Atlas dashboard.
2. Restore to a new cluster (do not overwrite the existing primary
   immediately — keep the corrupted state for forensics).
3. Verify data integrity: row counts on key collections,
   `findOne` smoke checks on User and Student.
4. Update `MONGODB_URI` on the Render service to point to the new
   cluster.
5. Redeploy the service. Run `/api/health` and Docs RAG smoke test.
6. Inform stakeholders that data older than the snapshot point is lost.
7. Re-enable any features that were disabled during the incident.

### Point-in-time restore (Atlas continuous backup, if enabled)

If Atlas continuous backup is enabled `[FILL]`, the DBA selects an exact
timestamp to restore to. Same overall procedure.

### Partial restore (single collection / user)

For accidental deletion of a small data set, restore the snapshot to a
side cluster, export the affected documents (e.g., via `mongoexport`),
and re-import to the live cluster. Document the action in the incident
log.

| Drill | Frequency | Owner |
|---|---|---|
| Restore to side cluster | Quarterly | DBA |

The current state is **no documented drill has been performed**. That is
the first action under this policy.

## What this plan does not protect against

- A logical bug that silently corrupts data over weeks (the snapshot may
  also carry the corruption). Mitigations: tests, audit trail, AuditLog
  collection (planned, SEC-16).
- A successful attack that reaches Atlas credentials and deletes both
  primary and snapshots. Mitigations: separate IAM principal for
  snapshot access; enable cross-region copy `[FILL]`.

## Vendor-side disaster

| Scenario | Action |
|---|---|
| Render Singapore down | Failover to a secondary Render region or alternative host `[FILL: documented runbook?]`; depend on Atlas remaining reachable from the new region |
| Atlas Ireland down | Wait for Atlas recovery; communicate with customers; consider a temporary read-only mirror in another region |
| OpenAI / Groq down | Docs RAG fallback already cascades; if both are down, AI features return graceful errors but the rest of the platform continues |
